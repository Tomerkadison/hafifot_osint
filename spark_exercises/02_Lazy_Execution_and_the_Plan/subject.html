חישוב עצל ותוכנית Catalyst

הכול עצל, ו-Catalyst משכתב את התוכנית שלכם לפני שנקראת שורה אחת.

1עצלות, אבל ברצינות

ב-Spark כל DataFrame עצל. כשאתם כותבים filter, select, withColumn — אתם רק מוסיפים שלב למתכון. שום מידע לא נקרא ושום חישוב לא רץ.

רק כשמגיע action (כמו show, count, write) — Spark לוקח את כל המתכון, מעביר אותו דרך ה-Catalyst optimizer, ורק אז מריץ תוכנית מיטבית על ה-cluster.

2מה Catalyst עושה עבורכם

Catalyst הוא מנוע אופטימיזציה שמשכתב את התוכנית לפני ההרצה:

  • Predicate pushdown — דוחף פילטרים קרוב ככל האפשר למקור, כדי לקרוא פחות מידע.
  • Projection pruning — קורא רק את העמודות שבאמת צריך (ראו ReadSchema ב-plan).
  • Filter combining — מאחד כמה filter לתנאי אחד.
  • Constant folding ועוד עשרות חוקים.

בגלל זה הסדר שבו אתם כותבים את הפעולות פחות קריטי מאשר ב-pandas — Catalyst יסדר מחדש בכל מקרה.

3לקרוא תוכנית עם explain()

df.explain() מדפיס את התוכנית הפיזית. קוראים אותה מלמטה למעלה: למטה ה-FileScan (קריאת הקובץ), ולמעלה הצעד האחרון.

df.explain(mode="extended") מראה את כל ארבעת השלבים: Parsed → Analyzed → Optimized → Physical. כך רואים בדיוק מה Catalyst שינה.

סימן ה-* ליד צעד (למשל *(1) Filter) אומר ש-Spark ייצר עבורו קוד Java מהיר (Whole-Stage Codegen).

4Polars lazy מול Spark lazy

⚡ Polars

  • pl.scan_csv(...) עצל
  • pl.read_csv(...) להוט
  • .collect() מריץ
  • .explain() מראה plan
  • רץ על מחשב אחד

🔥 Spark

  • כל DataFrame עצל
  • אין גרסה "להוטה"
  • show/count/collect מריצים
  • .explain() מראה plan
  • רץ במקביל על cluster
💡 ב-Polars אתם בוחרים lazy מול eager. ב-Spark תמיד lazy — וזה הופך את ה-Catalyst optimizer לחזק במיוחד.

5מתי לעשות cache()

כי הכול עצל, אם תשתמשו באותו DataFrame בכמה actions — Spark יחשב אותו מחדש כל פעם. אם זה יקר (join, aggregation), עשו df.cache() ואז action אחד כדי לממש אותו בזיכרון. מהפעם הבאה הוא יישלף מהזיכרון.

סיימתם? df.unpersist() כדי לשחרר. שימו לב: cache עצמו עצל — הוא נכנס לתוקף רק אחרי ה-action הבא.
