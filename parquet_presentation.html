הרעיון המרכזי: לשמור לפי עמודות

Row-based (CSV)

הנתונים נשמרים שורה אחרי שורה:

Dana30TLV
Avi25Haifa
Noa41TLV

כדי לקרוא רק את עמודת הגיל — קוראים הכל.

Columnar (Parquet)

כל עמודה נשמרת ברצף, אחת אחרי השנייה:

DanaAviNoa
302541
TLVHaifaTLV

רוצים רק גיל? קופצים ישר לבלוק הכחול ומדלגים על השאר.

בונוס ענק: דחיסה הרבה יותר טובה

כשכל הערכים באותו בלוק הם מאותו סוג ודומים זה לזה, קל מאוד לדחוס אותם:

  • Dictionary Encoding: במקום לשמור "Tel-Aviv" אלף פעמים, שומרים אותו פעם אחת במילון, ובנתונים רק את המספר 0.
  • RLE (Run Length): במקום 0,0,0,0,0 שומרים "0 כפול 5".
  • ולמעלה מזה: דחיסה כללית כמו Snappy או ZSTD על כל בלוק.

עמודת city — לפני ואחרי

// raw:
TLV, TLV, TLV, Haifa, TLV...

// dictionary:
dict = {0: "TLV", 1: "Haifa"}
data = 0,0,0,1,0...

// + RLE:
data = (0 ×3), (1 ×1), (0 ×1)

התוצאה בפועל: פי 5–10 קטן יותר מ-CSV

איך הקובץ בנוי מבפנים?

namecolumn chunk
agecolumn chunk
citycolumn chunk
Footer (Metadata) סכמה + מיקום של כל בלוק + סטטיסטיקות
  • Row Group: חבילה של שורות (בדרך כלל ~מיליון שורות או ~128MB). הקובץ מחולק לכמה כאלה.
  • Column Chunk: בתוך כל Row Group, כל עמודה שמורה כבלוק רציף משלה.
  • Footer: "תוכן העניינים" בסוף הקובץ. קוראים אותו ראשון — והוא מספר איפה כל דבר נמצא.

הנשק הסודי: סטטיסטיקות בכל Row Group

בזמן הכתיבה, פרקט שומר ב-Footer סטטיסטיקות על כל עמודה בכל Row Group:

  • Min / Max: הערך הקטן והגדול ביותר בבלוק.
  • Null Count: כמה ערכים ריקים יש.
  • Count: כמה שורות יש בבלוק.
למה זה גאוני?

אפשר לדעת מה יש בתוך בלוק בלי לקרוא אותו. כמו לקרוא את התווית על הקופסה במקום לפתוח אותה.

מה כתוב ב-Footer

Row Group 1 age: min=18, max=35

agenulls=0, count=1M

Row Group 2 age: min=36, max=59

agenulls=120, count=1M

Row Group 3 age: min=60, max=99

agenulls=5, count=1M

אופטימיזציה 1: Projection Pushdown

"תביא לי רק את העמודות שביקשתי"

אם השאילתה צריכה רק את age ו-city, המנוע קורא מהדיסק רק את ה-Column Chunks האלה ומדלג פיזית על כל השאר.

בטבלה עם 200 עמודות שבה צריך רק 3 — זה ההבדל בין לקרוא 100GB לבין לקרוא 1.5GB.

אופטימיזציה 2: Predicate Pushdown

"תביא לי רק שורות שעוברות את הפילטר"

השאילתה: age > 65

המנוע מסתכל על ה-Min/Max של כל Row Group ב-Footer:

  • Max של הבלוק הוא 35? אין סיכוי שיש שם age>65. מדלגים על הבלוק כולו בלי לקרוא אותו!
  • רק בלוקים שה-Min/Max שלהם חופף לפילטר נקראים בפועל.

WHERE age > 65

Row Group 1 min=18, max=35

Row Group 2 min=36, max=59

Row Group 3 min=60, max=99

age ✓ קוראים רק את זה

קראנו שליש מהקובץ במקום את כולו

ומי מחליט מה לדלג? כאן נכנס ה-Lazy

כדי לדעת שאפשר לקרוא רק 2 עמודות ובלוק אחד, המנוע חייב לראות את כל השאילתה מראש — לפני שהוא נוגע בדיסק.

  • Eager (כמו Pandas): כל שורת קוד רצה מיד. read_parquet קורא הכל, ורק אחר כך מסננים בזיכרון. מאוחר מדי לדלג.
  • Lazy (Polars / Spark): שום דבר לא רץ. כל פעולה רק נרשמת בתוכנית (Query Plan). הקריאה מהדיסק קורית רק בסוף — אחרי שהמנוע כבר יודע בדיוק מה צריך.
כותבים שאילתה: scan ← filter ← select
Optimizer מסתכל על התוכנית כולה
דוחף את ה-filter וה-select למטה, אל הקריאה
(זה ה-Pushdown — דחיפה במורד התוכנית)
קוראים מהדיסק רק מה שחייבים

איך זה נראה ב-Polars?

import polars as pl

# scan = lazy. שום דבר לא נקרא עדיין!
df = (
    pl.scan_parquet("huge_data.parquet")
      .filter(pl.col("age") > 65)   # Predicate Pushdown
      .select(["age", "city"])     # Projection Pushdown
      .collect()  # רק כאן הכל באמת רץ!
)

שימו לב להבדל: read_parquet = eager, קורא הכל מיד. scan_parquet = lazy, בונה תוכנית. אפשר להציץ בתוכנית האופטימלית עם df.explain() — תראו שם את ה-Pushdown בעצמכם.

ואיך זה נראה ב-Spark?

df = (
    spark.read.parquet("s3://bucket/huge_data/")
      .filter("age > 65")
      .select("age", "city")
)
# עדיין לא רץ כלום! Spark הוא lazy מטבעו.

df.show()  # פעולת Action — רק עכשיו הכל מתבצע
Transformations = תוכנית

filter, select, groupBy... רק נרשמים בגרף (DAG). ה-Catalyst Optimizer משכתב את התוכנית ודוחף פילטרים לרמת הקובץ.

Actions = ביצוע

show, collect, count, write... מפעילים את התוכנית. בודקים ב-df.explain(): חפשו PushedFilters בפלט.

החיבור המלא: צוות מנצח

Parquet

מארגן את הנתונים בעמודות, מחולק ל-Row Groups, ושומר סטטיסטיקות על הכל.

מאפשר לדלג

Lazy Evaluation

אוסף את כל השאילתה לתוכנית אחת לפני שנוגעים בדיסק, כך שיודעים מראש מה באמת צריך.

יודע על מה לדלג

Pushdown

האופטימייזר דוחף את ה-select (Projection) ואת ה-filter (Predicate) אל הקריאה עצמה.

מדלג בפועל

בלי lazy — אין pushdown. בלי המבנה של פרקט — אין על מה לדלג. ביחד: קוראים 2% מהקובץ במקום 100%.
