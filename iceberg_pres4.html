Apache Iceberg:
הבוס החדש של הדאטה

המדריך המלא למעבר מניהול קבצים לניהול טבלאות

מה זה בכלל Iceberg?

זה לא פורמט קובץ

הוא לא מחליף את Parquet. הקבצים שלכם נשארים Parquet, יעילים ודחוסים כמו תמיד.

זה פורמט טבלה

הוא ה"קלסר" שמנהל את הקבצים. הוא מגדיר מה נחשב טבלה, עוקב אחרי גרסאות ומאפשר טרנזקציות.

למה היינו צריכים את זה?

סיפור המקור: Netflix

נטפליקס ניהלו כמויות עצומות של דאטה על S3 והרגישו את הכאב של Hive:

  • חוסר אמינות: שאילתות נכשלו בזמן כתיבה.
  • ביצועים: סריקת אלפי קבצים לקחה נצח.
  • עדכונים: לשנות שורה אחת דרש שכתוב של הכל.

המעבר מניהול קבצים למסד נתונים

דמיינו אלפי קבצי Parquet מפוזרים ללא סדר. Iceberg הוא השכבה החכמה שמאחדת אותם לישות אחת אמינה, המאפשרת ביצוע טרנזקציות ACID מלאות ב-Data Lake.

הארכיטקטורה: ה"עוגה" של אייסברג

מטא-דאטה מול מניפסט

רמת הטבלה (Metadata): קובץ ה-JSON המרכזי שמנהל את ה-Schema וה-Snapshots של כל הטבלה.

רמת הקבצים (Manifest): רשימות המנהלות קבוצות קבצים, ושומרות סטטיסטיקות כמו טווחי ערכים (Min/Max) לכל קובץ כדי לייעל את השאילתות.

  • Metadata Layer: מנהלת את הטבלה ברמת המקרו (Schema וגרסאות).
  • Manifest Lists: מנהלת קבוצות קבצים ושומרת טווחי ערכים לסינון מהיר.
  • Data Files: קבצי ה-Parquet הפיזיים המאחסנים את הנתונים עצמם.

Snapshots: המפתח ל-ACID

גרסאות במקום דריסות

בכל פעם שכותבים מידע, נוצר Snapshot חדש. הישן לא נמחק מיידית.

  • Time Travel: שאילתה על גרסאות עבר בקלות.
  • Rollback: חזרה לגרסה קודמת בשנייה.
  • בידוד (Isolation): כתיבה לא מפריעה לקוראים.

זה לא עותק - זה מצביע (Pointer)

חשוב להבין: Snapshot הוא לא העתקה של המידע! זהו רק קובץ Metadata חדש וקטן שנוצר ברגע השינוי ומצביע על סט הקבצים העדכני. אין כאן שום העתקת בייטים פיזית של דאטה.

אסטרטגיה 1: Copy-on-Write (CoW)

הקונספט

צריכים לעדכן שורה? אייסברג קורא את כל קובץ ה-Parquet המקורי, משנה בזיכרון, וכותבת קובץ Parquet חדש ונקי.

למה שמישהו ירצה את זה?

  • קריאה במהירות מקסימלית: אין לקורא "עבודה" לעשות. הוא פשוט קורא קובץ Parquet מוכן.
  • פשטות: אין קבצי מחיקה מסובכים ברקע.

החיסרון: Write Amplification.

עדכון של 1KB בתוך קובץ של 100MB דורש כתיבה מחדש של כל ה-100MB.

אסטרטגיה 2: Merge-on-Read (MoR)

זו האסטרטגיה שתפתור לכם את כאב העדכונים התכופים.

מנגנון ה-Delete Files

במקום לשכתב קבצים, אייסברג כותב **"דף הערות" (Delete File)**:

  • הסתרת שורות: ה-Delete File אומר למנוע: "כשתקרא את קובץ A, תתעלם משורה 45".
  • כתיבה מהירה: מוסיפים רק את השינוי הקטן, לא נוגעים ב-200GB הקיימים.

מיזוג בזמן אמת

התהליך מפריד בין הכתיבה לקריאה. הכתיבה מהירה (רק דלתא), והקריאה מבצעת 'Merge' בין הנתונים למחיקות. זהו פתרון מושלם למערכות שמתעדכנות בקצב גבוה.

מנגנון ה-Update: מחק + הוסף

מחזור החיים של שורה מעודכנת

במקום לשנות שורה במקומה, Iceberg מבצע פעולה כפולה: רישום השורה הישנה בקובץ מחיקות (Delete) וכתיבת השורה החדשה בקובץ נתונים טרי. השילוב ביניהם מציג למשתמש את המידע העדכני.

פעולת ה-Update ב-Iceberg (במצב MoR) היא שילוב של שני קבצים חדשים:

  • Delete File: מסמן את השורה הישנה כ"מחוקה".
  • New Data File: מכיל את השורה החדשה (המעודכנת).
הקסם קורה בקריאה

מנוע השאילתות קורא את שניהם ומבצע Merge בזמן אמת. המשתמש רואה רק שורה אחת מעודכנת.

המלכוד: מוות ב-1000 חתכים

Small Files Problem

בכל דקה שאתם מעדכנים שורה ב-MoR, נוצר קובץ Delete חדש וקטן.

למה זה מאט אתכם?

כשיש 10,000 קבצי "הערות" (Delete Files), המנוע חייב לקרוא את כולם בזמן הקריאה כדי לדעת מה לסנן.

  • זמן שאילתה עולה: המנוע עושה Merge בין אלפי קבצים קטנים בזמן אמת.
  • ניצול CPU גבוה: ה-Overhead של ניהול המטא-דאטה הופך לצוואר בקבוק.

התחזוקה: Compaction

ה"מנקה" של אייסברג

Compaction הוא תהליך שרץ ברקע (או מתוזמן ב-Foundry) ותפקידו:

  • איחוד קבצים: לוקח המון קבצים קטנים והופך אותם לקובץ Parquet אחד גדול ואופטימלי.
  • מימוש מחיקות: הוא "שורף" את ה-Delete Files לתוך קובץ הנתונים החדש. עכשיו כבר לא צריך את ה-Delete files בזיכרון.
בפלנטיר: Maintenance Jobs

אל תשכחו להפעיל את ה-Dataset Optimization ב-Settings!

סיכום אסטרטגיות עדכון

מאפיין Copy-on-Write (CoW) Merge-on-Read (MoR)
מהירות כתיבה איטית (שכתוב הכל) מהירה (כתיבת דלתא בלבד)
מהירות קריאה מקסימלית (קבצים נקיים) איטית יותר (Merge בזמן אמת)
שימוש אופטימלי טבלאות ענק שמתעדכנות פעם ביום עדכונים תכופים (כמו המקרה שלכם)

השוואה: Hive מול Iceberg

יכולת Hive (השיטה הישנה) Iceberg (השיטה החדשה)
טרנזקציות ACID אין מלא וברמת השורה
עדכון שורה בודדת שכתוב מחיצה שלמה יעיל ומהיר (MoR)
Time Travel לא קיים מובנה באמצעות Snapshots
Schema Evolution שובר טבלאות לעיתים קרובות בטוח וגמיש (Add/Drop/Rename)

איך כותבים את זה ב-Foundry?

@incremental()
@transform_df(
    output=Output("..."),
    input_df=Input("...")
)
def compute(ctx, input_df, output):
    if ctx.is_incremental:
        # הקסם שמאפשר עדכון שורות ללא דריסה:
        output.set_mode("modify")
        output.write_dataframe(input_df)
    else:
        output.write_dataframe(input_df)

שימו לב: ללא ה-set_mode("modify"), המנוע ינסה לבצע Append או Overwrite רגיל, ולא ינצל את יכולות ה-MoR של Iceberg לעדכונים כירורגיים.
